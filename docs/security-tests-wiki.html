🍪 Bloque 1 — Cookies de sesión
COOKIE-SECURE Cookie flag: Secure Riesgo MEDIO ▶

¿Qué verifica?

Comprueba que todas las cookies en la respuesta HTTP tengan el atributo Secure. Este flag instruye al navegador a enviar la cookie únicamente sobre conexiones HTTPS cifradas.

¿Por qué importa?

Sin el flag Secure, si el usuario visita accidentalmente la versión HTTP del sitio (por un enlace, redirección maliciosa o ataque SSL stripping), el navegador enviará la cookie en claro. Un atacante en la misma red (Wi-Fi pública, MITM) puede capturarla y suplantar la sesión.

Resultados posibles

PASS Todas las cookies tienen el atributo Secure
FAIL Al menos una cookie no tiene el flag Secure

Remediación

# nginx.conf — agregar a la directiva de cookies vía proxy_cookie_flags
proxy_cookie_flags ~ secure samesite=lax;

# O con add_header para aplicar a todas las cookies del proxy:
proxy_cookie_path / "/; Secure; SameSite=Lax";
# httpd.conf o .htaccess — requiere mod_headers
Header always edit Set-Cookie ^(.*)$ "$1; Secure"
# conf/context.xml
<CookieProcessor sameSiteCookies="lax" />

# O en web.xml de la aplicación:
<session-config>
  <cookie-config>
    <secure>true</secure>
  </cookie-config>
</session-config>
# php.ini
session.cookie_secure = 1

# O en código PHP:
ini_set('session.cookie_secure', 1);
session_set_cookie_params(['secure' => true]);
// Express + express-session
app.use(session({
  cookie: {
    secure: true,   // requiere HTTPS
    httpOnly: true,
    sameSite: 'lax'
  }
}));
COOKIE-HTTPONLY Cookie flag: HttpOnly Riesgo ALTO ▶

¿Qué verifica?

Comprueba que la cookie de sesión principal tenga el atributo HttpOnly. Este flag impide que el código JavaScript pueda leer la cookie mediante document.cookie.

¿Por qué importa?

Es la defensa principal contra ataques XSS (Cross-Site Scripting). Si un atacante logra inyectar JavaScript en la página, sin HttpOnly puede robar la cookie de sesión directamente. Con HttpOnly, aunque haya XSS, la cookie permanece inaccesible desde el script.

Resultados posibles

PASS La cookie de sesión tiene el atributo HttpOnly
FAIL La cookie de sesión no tiene HttpOnly
SKIP No se encontró cookie de sesión en la raíz del dominio

Remediación

proxy_cookie_flags ~ secure httponly samesite=lax;
# O editando el Set-Cookie directamente:
proxy_cookie_path / "/; HttpOnly; Secure; SameSite=Lax";
Header always edit Set-Cookie ^(.*)$ "$1; HttpOnly; Secure"
<session-config>
  <cookie-config>
    <http-only>true</http-only>
    <secure>true</secure>
  </cookie-config>
</session-config>
# php.ini
session.cookie_httponly = 1
session.cookie_secure = 1
cookie: { httpOnly: true, secure: true, sameSite: 'lax' }
# config.php de Moodle
$CFG->cookiesecure = true;
$CFG->cookiehttponly = true;
COOKIE-SAMESITE Cookie flag: SameSite Riesgo MEDIO ▶

¿Qué verifica?

Verifica que las cookies tengan el atributo SameSite con valor Lax o Strict. Este atributo controla cuándo el navegador envía cookies en peticiones cross-site.

¿Por qué importa?

Protege contra ataques CSRF (Cross-Site Request Forgery). Con SameSite=Strict, las cookies nunca se envían en peticiones originadas desde otro dominio. Con Lax (recomendado), solo se envían en navegación top-level (links), no en peticiones de recursos embebidos.

Valores de SameSite

  • Strict: Máxima protección. La cookie nunca se envía cross-site. Puede causar problemas de UX (ej: al llegar desde un email).
  • Lax: Equilibrio recomendado. Bloquea peticiones POST, iframes, XHR cross-site. Permite navegación normal.
  • None: Sin protección. Requiere también Secure. Necesario para escenarios de SSO legítimos.

Remediación

proxy_cookie_flags ~ secure httponly samesite=lax;
Header always edit Set-Cookie ^(.*)$ "$1; SameSite=Lax"
session.cookie_samesite = "Lax"  # PHP 7.3+
cookie: { sameSite: 'lax' }
COOKIE-PATH Cookie attribute: Path Riesgo BAJO ▶

¿Qué verifica?

Comprueba que las cookies tengan definido el atributo Path. Sin él, la cookie se envía en cualquier ruta del dominio, incluyendo sub-aplicaciones que no deberían tener acceso a ella.

Remediación

Header always edit Set-Cookie ^(.*)$ "$1; Path=/app; SameSite=Lax"
session.cookie_path = "/app"
🔒 Bloque 2 — Transporte y TLS
TLS-HTTP-TO-HTTPS HTTP → HTTPS redirect 301/302 Riesgo MEDIO ▶

¿Qué verifica?

Realiza una petición HTTP al puerto 80 y verifica que el servidor responda con un redirect 301 o 302 hacia la URL HTTPS equivalente.

¿Por qué importa?

Sin redirect automático, los usuarios que accedan por HTTP navegan en claro. Combinado con SSL stripping, un atacante puede interceptar todas las comunicaciones incluso si el sitio soporta HTTPS.

Remediación

server {
    listen 80;
    server_name ejemplo.com;
    return 301 https://$host$request_uri;
}
<VirtualHost *:80>
    ServerName ejemplo.com
    Redirect permanent / https://ejemplo.com/
</VirtualHost>

# O con mod_rewrite:
RewriteEngine On
RewriteCond %{HTTPS} off
RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [R=301,L]
<!-- conf/web.xml — security-constraint -->
<security-constraint>
  <web-resource-collection>
    <web-resource-name>Redirect HTTP</web-resource-name>
    <url-pattern>/*</url-pattern>
  </web-resource-collection>
  <user-data-constraint>
    <transport-guarantee>CONFIDENTIAL</transport-guarantee>
  </user-data-constraint>
</security-constraint>
<!-- web.config -->
<rewrite>
  <rules>
    <rule name="HTTP to HTTPS" stopProcessing="true">
      <match url="(.*)" />
      <conditions>
        <add input="{HTTPS}" pattern="off" />
      </conditions>
      <action type="Redirect" url="https://{HTTP_HOST}/{R:1}" redirectType="Permanent" />
    </rule>
  </rules>
</rewrite>
TLS-HSTS HSTS — Strict-Transport-Security Riesgo ALTO ▶

¿Qué verifica?

Comprueba que la respuesta incluya el header Strict-Transport-Security con un max-age de al menos 31536000 segundos (1 año).

¿Por qué importa?

HSTS le indica al navegador que nunca acceda al dominio por HTTP, aunque el usuario lo escriba manualmente o haga clic en un enlace HTTP. Después del primer acceso HTTPS, el navegador memoriza esta instrucción y bloquea conexiones no cifradas automáticamente, eliminando la ventana de ataque SSL stripping.

Resultados posibles

PASS Header presente con max-age ≥ 31536000
WARN Header presente pero max-age < 31536000
FAIL Header ausente

Remediación

# Dentro del bloque server { } HTTPS
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
💡 Usa always para que el header se incluya también en respuestas de error (4xx, 5xx).
# Dentro del VirtualHost HTTPS
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"
<!-- web.xml de la aplicación -->
<filter>
  <filter-name>httpHeaderSecurity</filter-name>
  <filter-class>org.apache.catalina.filters.HttpHeaderSecurityFilter</filter-class>
  <init-param>
    <param-name>hstsEnabled</param-name>
    <param-value>true</param-value>
  </init-param>
  <init-param>
    <param-name>hstsMaxAgeSeconds</param-name>
    <param-value>31536000</param-value>
  </init-param>
</filter>
<httpProtocol>
  <customHeaders>
    <add name="Strict-Transport-Security" value="max-age=31536000; includeSubDomains" />
  </customHeaders>
</httpProtocol>
TLS-10-DISABLED TLS 1.0 deshabilitado Riesgo ALTO ▶

¿Qué verifica?

Intenta establecer una conexión TLS 1.0 con el servidor. Si el servidor la acepta, el test falla.

¿Por qué importa?

TLS 1.0 (1999) tiene vulnerabilidades conocidas como POODLE y BEAST que permiten descifrar tráfico cifrado. PCI-DSS y muchos marcos de cumplimiento exigen que TLS 1.0 esté deshabilitado desde 2018.

Remediación

ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256;
ssl_prefer_server_ciphers off;
SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1
SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256
SSLHonorCipherOrder off
<!-- server.xml -->
<Connector port="443" protocol="org.apache.coyote.http11.Http11NioProtocol"
           SSLEnabled="true">
  <SSLHostConfig protocols="TLSv1.2+TLSv1.3">
    <Certificate certificateKeystoreFile="..." />
  </SSLHostConfig>
</Connector>
TLS-11-DISABLED TLS 1.1 deshabilitado Riesgo MEDIO ▶

¿Qué verifica?

Intenta establecer una conexión TLS 1.1 con el servidor. TLS 1.1 fue oficialmente deprecado por la IETF en 2021 (RFC 8996).

Remediación

Misma configuración que TLS-10-DISABLED. Deshabilitar TLS 1.0 y 1.1 en el mismo bloque de configuración dejando solo TLS 1.2 y TLS 1.3.

💡 TLS 1.3 es el estándar recomendado actualmente. Ofrece mejor rendimiento (1-RTT handshake) y seguridad mejorada.
TLS-CERT-VALIDITY Certificado SSL vigente Riesgo CRÍTICO ▶

¿Qué verifica?

Conecta al servidor con openssl s_client y lee la fecha de expiración del certificado X.509. Calcula los días restantes hasta la expiración.

Resultados posibles

PASS Certificado vigente (más de 30 días)
WARN Expira en ≤ 30 días — renovar pronto
FAIL Expira en ≤ 7 días — CRÍTICO
SKIP No se pudo leer el certificado

Monitoreo y renovación

# Renovar todos los certificados vencidos o próximos a vencer
sudo certbot renew --dry-run   # Prueba sin aplicar
sudo certbot renew             # Renovación real

# Ver certificados gestionados
sudo certbot certificates
# Verificar fecha de expiración
echo | openssl s_client -connect ejemplo.com:443 -servername ejemplo.com 2>/dev/null \
  | openssl x509 -noout -dates

# Días restantes
echo | openssl s_client -connect ejemplo.com:443 2>/dev/null \
  | openssl x509 -noout -enddate \
  | cut -d= -f2 \
  | xargs -I{} date -d "{}" +%s \
  | xargs -I{} bash -c 'echo $(( ({} - $(date +%s)) / 86400 )) días'
# /etc/cron.d/ssl-check — alerta 30 días antes
0 8 * * 1 root /usr/local/bin/check-ssl.sh ejemplo.com admin@empresa.com
#!/bin/bash — check-ssl.sh
DOMAIN=$1; EMAIL=$2; DAYS=30
EXPIRY=$(echo | openssl s_client -connect ${DOMAIN}:443 2>/dev/null \
  | openssl x509 -noout -enddate 2>/dev/null | cut -d= -f2)
LEFT=$(( ($(date -d "$EXPIRY" +%s) - $(date +%s)) / 86400 ))
[[ $LEFT -le $DAYS ]] && echo "SSL de $DOMAIN expira en $LEFT días" | mail -s "⚠️ SSL Alert" $EMAIL
📋 Bloque 3 — Cabeceras de seguridad HTTP
HEADER-X-FRAME-OPTIONS X-Frame-Options (anti-clickjacking) Riesgo ALTO ▶

¿Qué verifica?

Comprueba que la respuesta incluya el header X-Frame-Options. Este header controla si el navegador permite mostrar la página dentro de un <iframe>, <frame> o <object>.

¿Por qué importa?

Sin este header, un atacante puede embeber tu aplicación en un iframe en su web maliciosa y usar técnicas de clickjacking: superponer capas invisibles para que el usuario haga clic en botones de tu app sin saberlo (cambiar contraseña, confirmar transferencias, etc.).

Valores recomendados

  • DENY: Nunca puede mostrarse en un frame (máxima protección)
  • SAMEORIGIN: Solo puede mostrarse en frames del mismo origen (recomendado para apps con iframes propios)

Remediación

add_header X-Frame-Options "SAMEORIGIN" always;
Header always set X-Frame-Options "SAMEORIGIN"
<!-- HttpHeaderSecurityFilter en web.xml -->
<init-param>
  <param-name>antiClickJackingEnabled</param-name>
  <param-value>true</param-value>
</init-param>
<init-param>
  <param-name>antiClickJackingOption</param-name>
  <param-value>SAMEORIGIN</param-value>
</init-param>
<add name="X-Frame-Options" value="SAMEORIGIN" />
// Con Helmet.js
app.use(helmet.frameguard({ action: 'sameorigin' }));
💡 En CSP moderno puedes usar frame-ancestors 'self' como alternativa más flexible a X-Frame-Options.
HEADER-X-CONTENT-TYPE-OPTIONS X-Content-Type-Options: nosniff Riesgo MEDIO ▶

¿Qué verifica?

Verifica que la respuesta incluya X-Content-Type-Options: nosniff. Este header impide que el navegador intente adivinar el tipo MIME de un recurso ignorando el Content-Type declarado.

¿Por qué importa?

Sin este header, si un atacante logra subir un archivo malicioso (ej: un HTML disfrazado de imagen), el navegador puede ejecutarlo como HTML aunque el servidor lo sirva como image/png. Esto puede llevar a XSS.

Remediación

add_header X-Content-Type-Options "nosniff" always;
Header always set X-Content-Type-Options "nosniff"
app.use(helmet.noSniff());
HEADER-CSP Content-Security-Policy (CSP) Riesgo ALTO ▶

¿Qué verifica?

Verifica que la respuesta incluya el header Content-Security-Policy. También detecta si contiene unsafe-eval, que debilita la política.

¿Por qué importa?

CSP es la defensa en profundidad más poderosa contra XSS. Define desde qué orígenes puede cargar el navegador scripts, estilos, imágenes, fuentes, etc. Un CSP bien configurado bloquea la ejecución de scripts inyectados aunque el atacante logre inyectarlos.

CSP mínimo recomendado para Moodle/PHP

# CSP básico — ajustar según los recursos reales del sitio
add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self' 'unsafe-inline';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'self';
  base-uri 'self';
  form-action 'self';
" always;
Header always set Content-Security-Policy \
  "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; frame-ancestors 'self';"
# config.php — Moodle tiene soporte nativo de CSP
$CFG->additionalhtmlhead = '<meta http-equiv="Content-Security-Policy"
  content="default-src \'self\'; script-src \'self\' \'unsafe-inline\' \'unsafe-eval\';
  style-src \'self\' \'unsafe-inline\'; img-src \'self\' data: https:;">';
⚠️ Moodle usa eval() en algunas funcionalidades JavaScript. Se necesita unsafe-eval temporalmente mientras se migran esos módulos.
app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"],
    styleSrc: ["'self'", "'unsafe-inline'"],
    imgSrc: ["'self'", "data:", "https:"],
    frameAncestors: ["'self'"],
  }
}));
# Primero despliega en modo Report-Only para ver qué se rompería
add_header Content-Security-Policy-Report-Only "
  default-src 'self';
  report-uri /csp-report-endpoint;
" always;
💡 Usa report-uri.com para capturar y analizar los reportes CSP antes de habilitar la política completa.
HEADER-REFERRER-POLICY Referrer-Policy Riesgo BAJO ▶

¿Qué verifica?

Verifica que la respuesta incluya el header Referrer-Policy. Este header controla cuánta información del Referer se envía al navegar a otro sitio.

¿Por qué importa?

Sin este header, al hacer clic en un enlace hacia un sitio externo, el navegador envía la URL completa de la página actual en el header Referer. Esto puede filtrar tokens en URLs, IDs de sesión o rutas internas a terceros (analítica, CDNs, etc.).

Remediación

add_header Referrer-Policy "strict-origin-when-cross-origin" always;
Header always set Referrer-Policy "strict-origin-when-cross-origin"
HEADER-PERMISSIONS-POLICY Permissions-Policy Riesgo BAJO ▶

¿Qué verifica?

Verifica que la respuesta incluya el header Permissions-Policy (anteriormente Feature-Policy). Controla qué APIs del navegador (cámara, micrófono, geolocalización) pueden usar la página o sus iframes.

Remediación

add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;
Header always set Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()"
💡 () significa que la feature está deshabilitada para todos. Ajusta si tu aplicación necesita alguna de estas capacidades.
🕵️ Bloque 4 — Fuga de información
INFOLEAK-SERVER-HEADER Server header oculto (sin versión) Riesgo MEDIO ▶

¿Qué verifica?

Inspecciona el header Server de la respuesta. Si contiene un número de versión (ej: nginx/1.18.0, Apache/2.4.41), el test falla.

¿Por qué importa?

Revelar la versión exacta del servidor facilita a los atacantes identificar versiones vulnerables y usar exploits específicos sin necesidad de fingerprinting adicional.

Remediación

# nginx.conf — bloque http {}
server_tokens off;
# httpd.conf o apache2.conf
ServerTokens Prod
ServerSignature Off
<!-- server.xml — atributo server del Connector -->
<Connector port="8080" server="Apache" ... />

<!-- Para ocultarlo completamente, usar un filtro en web.xml -->
<filter>
  <filter-name>httpHeaderSecurity</filter-name>
  <filter-class>org.apache.catalina.filters.HttpHeaderSecurityFilter</filter-class>
</filter>
INFOLEAK-X-POWERED-BY X-Powered-By ausente Riesgo MEDIO ▶

¿Qué verifica?

Verifica que no exista el header X-Powered-By. Este header es enviado automáticamente por PHP, Express y otros frameworks revelando el stack tecnológico (ej: PHP/8.1.2).

Remediación

# Eliminar el header si viene del backend
proxy_hide_header X-Powered-By;
fastcgi_hide_header X-Powered-By;
# php.ini
expose_php = Off
// Express deshabilita por defecto, o con Helmet:
app.disable('x-powered-by');
// O con Helmet:
app.use(helmet.hidePoweredBy());
Header always unset X-Powered-By
Header always unset X-Generator
INFOLEAK-ASP-NET-VERSION X-AspNet-Version ausente Riesgo MEDIO ▶

¿Qué verifica?

Verifica que no existan los headers X-AspNet-Version o X-AspNetMvc-Version, emitidos automáticamente por aplicaciones ASP.NET.

Remediación

<system.web>
  <httpRuntime enableVersionHeader="false" />
</system.web>

<system.webServer>
  <httpProtocol>
    <customHeaders>
      <remove name="X-Powered-By" />
      <remove name="X-AspNet-Version" />
    </customHeaders>
  </httpProtocol>
</system.webServer>
// Program.cs (.NET 6+)
app.Use(async (context, next) => {
    context.Response.Headers.Remove("X-AspNet-Version");
    context.Response.Headers.Remove("X-AspNetMvc-Version");
    await next();
});
⚙️ Bloque 5 — Configuración del servidor
SERVERCFG-CORS-WILDCARD CORS — sin wildcard Access-Control-Allow-Origin Riesgo ALTO ▶

¿Qué verifica?

Revisa el header Access-Control-Allow-Origin. Si el valor es *, cualquier sitio externo puede hacer peticiones AJAX a esta API y leer la respuesta desde JavaScript, incluyendo datos del usuario autenticado.

Remediación

# Restringir a un origen específico
add_header Access-Control-Allow-Origin "https://app.miempresa.com" always;

# Para múltiples orígenes, usar variable:
map $http_origin $cors_origin {
    "https://app.miempresa.com" $http_origin;
    "https://admin.miempresa.com" $http_origin;
    default "";
}
add_header Access-Control-Allow-Origin $cors_origin always;
SetEnvIf Origin "^https://(app|admin)\.miempresa\.com$" ORIGIN=$0
Header always set Access-Control-Allow-Origin "%{ORIGIN}e" env=ORIGIN
const cors = require('cors');
app.use(cors({
  origin: ['https://app.miempresa.com', 'https://admin.miempresa.com'],
  credentials: true
}));
⚠️ Si el endpoint es público y no requiere autenticación, * puede ser aceptable. El riesgo es crítico solo en APIs que devuelven datos de sesión o requieren cookies.
SERVERCFG-HTTP-TRACE HTTP TRACE deshabilitado Riesgo MEDIO ▶

¿Qué verifica?

Envía una petición TRACE al servidor y verifica la respuesta. Si responde con HTTP 200, el método está activo.

¿Por qué importa?

El método TRACE está diseñado para diagnóstico: el servidor devuelve la petición tal como la recibió. Esto habilita el ataque XST (Cross-Site Tracing): un script malicioso puede usar TRACE para leer headers que normalmente son inaccesibles (incluyendo cookies HttpOnly en algunos navegadores antiguos).

Remediación

# Rechazar método TRACE
if ($request_method = TRACE) {
    return 405;
}
TraceEnable off
<!-- web.xml de la aplicación -->
<security-constraint>
  <web-resource-collection>
    <web-resource-name>Disable TRACE</web-resource-name>
    <url-pattern>/*</url-pattern>
    <http-method>TRACE</http-method>
  </web-resource-collection>
  <auth-constraint />
</security-constraint>
SERVERCFG-CACHE-CONTROL Cache-Control adecuado Riesgo MEDIO ▶

¿Qué verifica?

Verifica que el header Cache-Control esté presente y contenga directivas como no-store, no-cache o private.

¿Por qué importa?

Sin directivas de caché apropiadas, el navegador puede almacenar páginas que contienen datos sensibles (expedientes, datos personales, reportes financieros). Si el usuario usa un equipo compartido, la siguiente persona puede acceder al historial del navegador o a la caché y ver esos datos sin autenticarse.

Remediación

# Para rutas autenticadas / con datos sensibles:
location /app/ {
    add_header Cache-Control "no-store, no-cache, must-revalidate, private" always;
    add_header Pragma "no-cache" always;
}

# Para recursos estáticos (JS, CSS, imágenes) — caché agresiva:
location ~* \.(js|css|png|jpg|woff2)$ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}
<LocationMatch "^/app/">
    Header always set Cache-Control "no-store, no-cache, must-revalidate, private"
    Header always set Pragma "no-cache"
</LocationMatch>
header("Cache-Control: no-store, no-cache, must-revalidate, private");
header("Pragma: no-cache");
header("Expires: 0");
res.set('Cache-Control', 'no-store, no-cache, must-revalidate, private');
res.set('Pragma', 'no-cache');
💡 No apliques no-store a recursos estáticos (CSS, JS, imágenes). Solo a rutas que sirven contenido dinámico con datos del usuario.
